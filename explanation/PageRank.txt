This code block implements one iteration of the PageRank power method, computing the next PageRank vector r_new from the current r. It follows the Google matrix formula: G = (1 - p) * A_hat + (p / N) * e * e^T, where A_hat handles dangling nodes, p is the teleportation probability, and e is the all-ones vector. The steps avoid materializing the dense matrix G for efficiency.

Block 1: Sparse Link-Following Step
Computes r_new = A * r, where A is the column-stochastic sparse adjacency matrix (CSR format).
This represents the link-following part: each node's new score is the sum of incoming scores from its neighbors, weighted by the link probabilities.
self.A.dot(r) performs a sparse matrix-vector multiplication (O(nnz), where nnz is non-zero entries), efficient for large graphs.

Block 2: Dangling Node Correction
Handles dangling nodes (nodes with no outgoing links): Their "lost" PageRank mass (sum of scores on dangling nodes) is redistributed uniformly across all nodes.
dangling_sum is the total score on dangling nodes; adds dangling_sum / N to each element of r_new.
This implements the A_hat adjustment in the Google matrix, ensuring the matrix remains stochastic (columns sum to 1).

Block 3: Teleportation
Applies the teleportation (damping) factor: alpha = 1 - p (link-following weight), p is the teleportation probability (e.g., 0.15).
alpha * r_new scales the link-following result; p / N adds uniform probability to jump to any node.
This prevents rank sinks and ensures convergence; equivalent to adding (p / N) * e to the matrix-vector product.

Block 4: Convergence Check
Measures convergence using the L1 norm of the difference: residual = ||r_new - r||_1.
If residual < tol (tolerance, e.g., 1e-8), the algorithm has converged (PageRank vector is stable).
Appends to residuals list for tracking; used to decide if to continue iterating.

In the run method of the PageRankEngine class, r represents the PageRank vector (also called the PageRank scores or stationary distribution). It calculates the relative importance or authority of each node in the graph based on the link structure and the teleportation probability p.

What r Represents
Type: np.ndarray of shape (N,), where N is the number of nodes.
Values: Probabilities that sum to 1.0 (a proper probability distribution).
Interpretation: r[i] is the PageRank score for node i, indicating how likely a random surfer is to visit that node after many steps.
How r Is Computed
r is iteratively updated using power iteration to solve the equation r = G * r, where G is the Google matrix (not explicitly built due to sparsity). Each iteration refines r until convergence:

Initialization: Starts as a uniform distribution (r = [1/N, 1/N, ..., 1/N]) or from a custom r0.
Iteration Loop:
r_new = A.dot(r): Propagates scores along links.
r_new += (dangling_sum / N): Redistributes mass from dangling nodes uniformly.
r_new = alpha * r_new + (p / N): Applies teleportation (random jumps).
r = r_new: Updates r for the next iteration.
Convergence: Stops when the change in r (L1 residual) is below tol, or after max_iter iterations.
Final r: The converged PageRank scores, returned in PageRankResult.
Key Properties
Sums to 1: r.sum() ≈ 1.0.
Higher values: Indicate more authoritative/central nodes.
Influenced by p: Lower p emphasizes link structure; higher p makes scores more uniform.
In the returned PageRankResult, r is stored as scores, and you can access top-ranked nodes via result.top_k(). This vector is the core output of PageRank, used for ranking web pages, nodes in networks, or entities in knowledge graphs.

Why use sp.eye(3)
sp.eye(3) creates a sparse identity matrix instead of np.eye(3) which creates a dense one.

The difference
np.eye(3) — dense:


[[1., 0., 0.],
 [0., 1., 0.],
 [0., 0., 1.]]
Stores all 9 values in memory, including the 6 zeros.

sp.eye(3, format="csr") — sparse:


Only stores: [(0,0)=1, (1,1)=1, (2,2)=1]
Stores only the 3 non-zero values.

Why it matters here
A_hat is already a sparse matrix. If you subtract a dense I from it:

The result B becomes dense
All sparsity of A_hat is destroyed
spsolve loses its efficiency advantage
By using sp.eye, B stays sparse, and spsolve can exploit that sparsity for the O(N^1.5)–O(N^2) solve instead of O(N^3).

At scale
For a real web graph with N = 10,000 nodes:

Matrix	Storage
Dense np.eye(10000)	800 MB
Sparse sp.eye(10000)	~0.24 MB (only N non-zeros)
The format="csr" specifically matches A_hat's format so the subtraction I - alpha*A_hat doesn't trigger a format conversion, keeping the operation efficient.

Yes. The choice is purely about efficiency, not correctness. Both give the same mathematical result.

Dense version (works, but slow)

I = np.eye(self.N)              # dense N×N
B = I - alpha * A_hat.toarray() # must convert A_hat to dense too
r = np.linalg.solve(B, rhs)     # dense solver
Correct answer, but:

Memory: O(N²) — 800 MB for N=10,000
Solve: O(N³)
Sparse version (what the code uses)

I = sp.eye(self.N, format="csr") # sparse
B = I - alpha * A_hat            # stays sparse
r = spla.spsolve(B, rhs)         # sparse solver
Same answer, but:

Memory: O(N) — only non-zeros stored
Solve: O(N^1.5)–O(N²)
Rule of thumb
Web/graph adjacency matrices are extremely sparse — a page with 10 links in a graph of 1 million nodes has a density of 0.000001%. Using dense matrices here would be like storing a map of the world just to mark 10 cities.

So yes, you can use dense — it's just impractical beyond toy-sized graphs (N > ~1,000).
